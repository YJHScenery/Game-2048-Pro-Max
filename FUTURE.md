
# FUTURE / 未来拓展规划

本文件基于 `docs/` 下的规划材料（排除 `GPT-5.2-Medium_Analysis_26-3-29-22-49.txt`）整理，目标是把“高维 2048”从可运行 Demo，推进到具备高性能计算/异构后端/高维可视化/可科研化包装的完整项目。

## 0. 项目愿景（定位）

将本项目定位为：

**高维 2048 = 高维张量计算与状态空间搜索的验证载体**

主线能力：

- **核心层**：高维棋盘/张量模型（C++20 + 模板元）
- **存储层**：稀疏/稠密自适应存储与布局优化（面向 GB 级规模）
- **计算层**：CPU / CUDA（可扩展到 CANN）统一抽象后端
- **关键难点**：超大规模状态的“可用可视化”（增量同步/压缩/LOD）
- **交互层**：Qt Quick 高维切片/编码可视化 + 性能监控
- **拔高项**：GPU 并行推演与 MCTS/启发式搜索（让 GPU 成为核心而非“玩具加速”）

## 1. 总体架构演进（模块化落地）

建议以“上层逻辑不关心后端/存储细节”为原则，逐步稳定接口。

### 1.1 核心接口（建议稳定为项目 ABI）

对外统一为一套“棋盘/张量操作”接口：

- 元信息：维度 `D`、边长 `L`、总元素数 `N=L^D`
- 访问：`get/set/is_zero`（坐标 → 值）
- 统计：`non_zero_count`、填充率 `fill_rate = non_zero_count / total_size`
- 规则：`slide_axis(axis, forward)` + 合并/补零/判停 + 随机生成新块
- 后端：`cpu/cuda/cann` tag 或策略类（编译期/运行期择一）

### 1.2 存储层：稀疏 ↔ 稠密自适应

依据规划材料的工程结论：

- **前期稀疏**：大量 0，适合块稀疏 + 哈希（坐标压缩 → 值）
- **后期稠密**：填充率上升后，稀疏结构维护成本反超，应一次性转为稠密（连续内存 + GPU 友好）

推荐策略：

- 每 `N` 步（如 4 步）或每次生成新块后计算一次 `fill_rate`
- `fill_rate < 20%` 保持稀疏，`fill_rate >= 20%~30%` 触发一次 **Sparse → Dense** 转换
- 转成稠密后 **锁定**（避免来回切换导致抖动与拷贝浪费）

### 1.3 模板元（TMP）深化：从“参数化”到“性能骨架”

将 TMP 用作编译期优化工具而非简单参数传递：

- **编译期 strides**：高维坐标↔线性索引展开，减少运行时循环与分支
- **轴/方向/类型/后端编译期派发**：消除大量 `if/switch`，让每个轴/方向实例化最优实现
- **Traits**：统一管理存储布局、对齐、是否启用 GPU、是否需要分片/分块
- **表达式/DSL（进阶）**：将 slide/merge/compress 等操作组合成链式表达，避免中间临时张量

### 1.4 计算层：让 GPU/CANN 变“核心引擎”

从“单步加速”升级为“并行状态空间搜索/推演”：

- 批量棋盘（BatchBoard）：一次 kernel 并行推演成千上万局
- 并行 rollout：MCTS / 随机模拟 / 启发式评估
- 并行展开：10 维下每步约 `2*D` 种动作，搜索 2~3 层分支爆炸，GPU 更有意义

工程取舍建议（来自文档策略）：

- **稀疏阶段**：优先 CPU（数据量小、结构复杂、GPU 稀疏代价高）
- **稠密阶段**：切换到 CUDA/CANN（连续内存、合并访存、吞吐最大化）

### 1.5 交互层：高维可视化不是“画出 10 维”

核心策略：**降维 + 切片 + 视觉通道编码**

- 2D/3D 切片：用滑块控制每个维度的切片坐标，只渲染当前切片
- 维度编码：颜色/透明度/大小/粒子密度映射高维统计量（稀疏度/数值强度等）
- 渐进式 LOD：只加载视口/焦点附近精细数据，其余区域低精度
- “工程加分项”：界面展示传输量、脏块数、帧率、耗时分布

## 2. 一年期路线图（12 个月，可直接用于开题/项目书）

### 阶段 A（1～3 月）：MVP 打通闭环（稠密 + 后端 + UI）

目标：最小可用系统，能玩、能交互、能演示。

- 稠密棋盘：`Eigen::Tensor`（或等价连续存储）
- 规则：滑动/合并/生成/判停
- 后端抽象：CPU + CUDA（可预留 CANN 接口）
- Qt Quick：基础 2D/切片展示，完成操作→计算→显示的数据流

产出：

- 可交互运行的高维 2048 原型
- CPU/CUDA 统一调用接口

### 阶段 B（4～6 月）：TMP 优化 + 稀疏/稠密自适应

目标：从“能用”到“现代 C++ + 高性能计算结构清晰”。

- TMP：编译期索引/strides/轴方向派发
- 存储：块稀疏（坐标压缩 + 哈希）与稠密双实现
- 切换：填充率阈值触发 Sparse→Dense，一次性转换并锁定
- 观测：性能面板（耗时、内存、填充率、切换时机）

产出：

- 自适应存储机制（可写“创新点”）
- 性能对比数据（稀疏 vs 稠密、CPU vs CUDA）

### 阶段 C（7～9 月）：GPU 并行搜索（MCTS/并行推演）

目标：让 GPU 成为核心竞争力。

- BatchBoard：批量棋盘管理（Host/Device 数据组织）
- CUDA 并行 rollout：一次 kernel 并行模拟大量局面
- 评估函数：空格数/平滑度/单调性等启发式（先简单、可持续迭代）
- 基准：CPU 单线程/多线程 vs GPU 吞吐，输出图表

产出：

- 高维 2048 自动求解/演示模式
- 可写实验章节的性能曲线

### 阶段 D（9～10 月）：Qt Quick 可视化增强与流畅度优化

目标：展示效果拉满、不卡顿。

- 高维切片 + 维度聚焦交互（滑块/下拉）
- 热力/颜色编码、合并/出现/滑动动画
- 批量渲染优化（避免大量 QML Item 导致 DrawCall 爆炸）
- 可选：Qt Quick 3D 进行 3D 切片展示

产出：

- 工业级演示界面（适合答辩/展出）

### 阶段 E（11～12 月）：光电/信号方向拓展与项目包装

目标：从“游戏 demo”升格为“高维数据计算与可视化系统”。

三选一落地路线：

- 路线 A（大创）：高维数据异构并行计算引擎（将 2048 内核抽象为通用算子框架）
- 路线 B（光赛）：高维/光场数据交互式可视化系统（重展示、重体验）
- 路线 C（毕设）：自适应存储 + 异构加速 + 可视化闭环（重完整度与实验）

产出：

- 论文/报告/申报书结构化材料
- 可复用到光场/光谱/波前等数据的“引擎 + UI”骨架

## 3. 关键风险与对策（提前写进计划）

- **风险：GB 级数据不可实时回传/绘制** → 对策：增量同步、脏块标记、LOD、多分辨率、只渲染焦点/视口
- **风险：GPU 架构差异导致编译失败/性能差** → 对策：将 `CMAKE_CUDA_ARCHITECTURES` 做成可配置；提供 CPU fallback
- **风险：稀疏结构在 GPU 上复杂** → 对策：稀疏阶段走 CPU，稠密阶段再上 GPU
- **风险：UI 卡顿** → 对策：减少 QML Item 数量，必要时用自定义渲染项批量绘制

## 4. 可量化验收指标（建议）

- 支持维度/边长组合：至少覆盖 2D/4D/6D（可逐步扩到 10D）
- 存储：稀疏阶段内存占用显著低于稠密；切换后吞吐显著提升
- 计算：GPU 并行 rollout 相对 CPU 提升一个数量级（以“每秒推演局面数”为指标）
- 交互：切片切换与操作响应稳定（目标 30/60 FPS，视棋盘规模分级）
