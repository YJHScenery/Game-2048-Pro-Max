稀疏矩阵与稠密矩阵切换思路
下面给你一套工程上可直接落地、和你现有 TMP + Eigen + CUDA/CANN 完全兼容的思路，不玩虚的，从数据结构、切换策略、内存、GPU 适配、性能五个维度讲清楚。
你遇到的场景非常典型：
前期：大量 0 → 稀疏矩阵优势极大
后期：几乎填满 → 稀疏反而变慢、变复杂
需求：运行时自动在稀疏 ↔ 稠密之间无缝切换
核心总思路
稀疏与稠密共用同一套逻辑接口，底层存储用类型擦除或模板分派切换，
填充率达到阈值时，执行一次“格式转换 + 内存搬迁”，之后全程稠密。
不会破坏你现有的 2048 滑动、合并、随机生成逻辑。
一、第一步：统一上层接口（最关键）
不管稀疏还是稠密，对外暴露完全一致的访问函数：
// 公共接口
T get(Coord...);
void set(Coord..., T value);
bool is_zero(Coord...);
size_t non_zero_count();
void slide_axis(int axis, bool forward);
内部存储用两种方式：
DenseStorage：Eigen::Tensor / 裸数组
SparseStorage：CSR/COO 或哈希表坐标 → 值
上层逻辑完全不关心底层是啥，这是你 TMP 发挥的地方。
二、稀疏矩阵用什么格式最合适？
对你 10 维 2048 这种高维、规则网格、动态更新场景：
推荐：块稀疏 + 哈希表（坐标→值）
不使用传统 CSR/CSC（高维极难维护）
用 unordered_map<MultiKey, T> 只存非零块
MultiKey 可通过 TMP 把多维坐标编译期压缩成一个整数
优点：
插入、删除、随机访问极快
滑动合并逻辑几乎不用改
高维下唯一简单好用的稀疏结构
次选：Eigen Sparse
但 Eigen Sparse 对高维张量支持弱，更适合 2D。
你自己用 TMP 写一个高维哈希稀疏存储，反而更灵活、更快。
三、切换策略：什么时候从稀疏切稠密？
1. 阈值判断（最简单有效）
定义填充率：

\text{fill\_rate} = \frac{\text{non\_zero\_count}}{\text{total\_size}}
经验阈值：
fill_rate < 20% → 稀疏
fill_rate ≥ 20% ~ 30% → 切换稠密
高维 8¹⁰ 后期填充率会迅速飙升，切换一次就不再切回稀疏，避免反复转换。
2. 切换触发时机
不要每步都判断，太耗性能：
每 N 步判断一次（比如每 4 步）
或每次新块生成后判断
一旦切换，置一个 is_dense_mode 标志，永久锁定稠密
四、稀疏 → 稠密的转换过程（核心步骤）
切换是一次性 O(N) 开销，但后期收益巨大。
流程：
新建稠密张量（Eigen::Tensor 或连续数组）
遍历稀疏结构所有非零元素，填入稠密对应坐标
释放稀疏内存（重要，你是 GB 级数据）
后续所有操作直接走稠密逻辑
对外接口完全不变
用 TMP 可以做到：
编译期生成坐标展开代码
转换时无运行时开销
对 CUDA/CANN 友好（稠密内存连续）
五、模板元如何优雅支持双存储？
你可以用 tag dispatch + 模板特化，非常干净：
struct SparseTag {};
struct DenseTag {};

template<int Dim, int Size, typename T, typename StorageTag>
class Board;

// 稀疏特化
template<...>
class Board<..., SparseTag> {
    unordered_map<MultiKey, T> data_;
};

// 稠密特化
template<...>
class Board<..., DenseTag> {
    Eigen::Tensor<T, Dim> data_;
};
切换时：
构造稠密版
批量拷贝
析构稀疏版
用 unique_ptr<BaseBoard> 类型擦除持有
这样：
CUDA/CANN 后端只需要处理稠密连续内存
代码结构极度清晰
完全符合高性能计算规范
六、CUDA / CANN 如何适配？
GPU 对稀疏支持很麻烦，你的策略可以非常现实：
1. 稀疏阶段：CPU 计算
前期稀疏 → 反正数据小
CPU 足够快，不麻烦 GPU
2. 切换稠密后 → 再进 GPU/CANN
稠密内存连续
合并访存拉满
核函数最简单、最快
完全避开高维稀疏在 GPU 上的噩梦
这是工程最优解，比硬写 GPU 稀疏算子聪明得多。

切换稠密后：正常绘制整个网格
切换时加一个淡入动画，用户无感知
你甚至可以在界面显示：

填充率：XX.X%

这在毕设/大创里是肉眼可见的优化亮点。
